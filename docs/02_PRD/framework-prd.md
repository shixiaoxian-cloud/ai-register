# Framework PRD — ai-register
> 由 `ai-pm` 在 Scene 2-4 中持续补全。当前作为项目级产品框架入口。

## 1. Background & Value Scope
- **Current Iteration Theme**: 前端控制台产品化重构。
- **Target Users（当前假设）**:
  - 内部测试操作者：需要配置目标站点、启动验证流程、查看运行过程和判断保护是否触发。
  - 内部维护者：需要管理临时邮箱配置、检查报告产物、排查异常运行。
- **Problem Definition**:
  - 当前前端由原生 HTML 工具页组成，功能可用但产品感弱、模块分散、视觉和交互不统一。
  - 配置、运行、日志、报告、邮箱相关能力还没有形成统一后台体验，难以支持后续持续扩展。
  - 配置和产物目前仍然分散在多个 JSON 文件与目录里，难以支撑多站点、多方案、多任务和任务级下载。
- **Platform Direction**:
  - 产品后续将演进为自动化测试平台，支持针对不同站点制定不同测试方案并持续执行验证。
  - 因此信息架构不能只围绕“单次运行”，还需要支撑站点资源、方案资源和跨运行产物管理。
- **Value Anchor**:
  - 从“多个临时工具页 + 脚本入口”转为“一个统一的内部测试控制台”。
  - 让用户可以在单一控制台里完成站点管理、方案配置、执行验证、观察过程、排查问题和查看结果，不再依赖页面切换和命令记忆。
- **Scope Boundary**:
  - 本轮优先重构前端信息架构、交互结构、视觉语言和组件基础，并补齐本地 SQLite 平台存储层。
  - 默认复用现有 Node/Playwright 能力接口，不把本轮扩大成远程后端平台重写。
- [待定-001] 是否在本轮就纳入“多角色权限 / 多用户后台”能力；如果不纳入，则先按单管理员后台设计。

## 2. Critical User Journeys
### CUJ-01 配置并发起一次验证任务
- **Start**: 用户进入控制台首页，确认当前目标站点和邮箱配置状态。
- **Action Path**: 修改目标站点配置 → 选择运行模式 → 启动测试。
- **Outcome**: 控制台进入运行态，用户能持续看到任务状态和最新日志。

### CUJ-02 在运行过程中判断是否出现保护或阻断
- **Start**: 任务已启动，用户进入运行详情或实时日志区域。
- **Action Path**: 查看最新阶段、关键日志、系统提示、人工介入提示。
- **Outcome**: 用户快速判断当前是成功推进、验证码流程、保护挑战还是阻断。

### CUJ-03 在任务结束后查看结果与产物
- **Start**: 测试运行结束。
- **Action Path**: 查看运行结论 → 打开 HTML 报告 → 检查日志、截图、trace 或 token 产物状态 → 按任务下载归档包。
- **Outcome**: 用户完成验证闭环，能判断“保护是否触发、是否拦截、人工处理后能否继续”，并能把任务相关证据整体导出。

### CUJ-04 管理临时邮箱与相关配置
- **Start**: 用户需要使用或调试临时邮箱流程。
- **Action Path**: 查看当前临时邮箱配置 → 测试连接 → 启用/调整配置。
- **Outcome**: 用户确认邮箱服务可用，并能支撑自动化验证码流程。

### CUJ-05 排查失败任务
- **Start**: 某次任务失败或结果未明。
- **Action Path**: 进入运行详情 → 检查关键阶段、错误日志、洞察卡片和报告入口。
- **Outcome**: 用户知道失败发生在哪一阶段，下一步应查看什么证据或进行什么人工动作。

### CUJ-06 为某个站点制定测试方案
- **Start**: 用户需要针对新的目标站点建立可重复执行的测试配置。
- **Action Path**: 新增站点资源 → 配置基础信息与目标地址 → 选择或编辑测试方案 → 保存为可运行配置。
- **Outcome**: 平台形成“站点 + 方案”的可复用资源，为后续批量执行和持续验证打基础。

## 3. Screen Tree
### Proposed Screen Tree（Resource-Centric Operations Hub）
- **Overview / 总览**
  - 平台概览
  - 站点与方案健康度
  - 近期运行结果
  - 快捷入口
- **Config Center / 配置中心**
  - Sites / 站点管理
  - Plans / 测试方案管理
  - Target Profiles / 选择器与规则配置
  - Temp Mail / 邮箱与验证码配置
- **Runs / 运行中心**
  - 运行列表
  - 运行详情
  - 阶段追踪
  - 实时日志
  - 人工介入提示
- **Artifacts / 结果产物**
  - Reports / HTML 报告
  - Logs / 运行日志
  - Media / 截图与视频
  - Trace / 调试追踪
  - Tokens / 输出凭据
  - Task Bundles / 任务下载
- **System / 系统设置**
  - 默认运行偏好
  - 前端显示偏好
  - 连接与环境说明
- [待定-002] `Config Center` 是否作为一级聚合页，还是直接拆成 `Sites / Plans / Mail` 多个一级导航。
- [待定-003] `Target Profiles` 是作为方案的一部分内嵌编辑，还是保留为独立资源类型。

## 4. Entities
- **Site / 站点**
  - 描述一个被授权测试的目标站点及其基础信息。
- **Plan / 测试方案**
  - 描述某站点对应的测试流程、运行参数、邮箱策略和结果判定规则。
- **Target Profile / 站点画像**
  - 描述选择器、期望结果、挑战识别规则和可选字段填充逻辑。
- **Run / 运行记录**
  - 描述一次执行实例，包括状态、阶段、日志、最终结论和人工介入信息。
- **Task / 任务**
  - 描述一次由平台发起的聚合工作单元，可关联多个 case、run 和下载归档包。
- **Case / 用例实例**
  - 描述任务下的单个执行单元，用于承接具体运行与产物关联。
- **Artifact / 结果产物**
  - 描述某次运行产生的报告、日志、trace、截图、视频、token 等输出。
- **Mail Config / 邮箱配置**
  - 描述临时邮箱服务或 IMAP 服务的接入方式与运行约束。
- [待定-004] `Plan` 与 `Target Profile` 是否合并为单一“测试方案”实体；如果拆分，需要明确谁负责复用、谁负责执行。
