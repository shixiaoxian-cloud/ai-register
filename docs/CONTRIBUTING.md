# 文档贡献指南

## 文档组织原则

本项目的所有技术文档统一存放在 `docs/` 目录下，按以下分类组织：

### 1. guides/ - 使用指南
面向用户的操作指南、配置说明和最佳实践。

**适合放在这里的内容：**
- 功能使用教程
- 配置步骤说明
- 工具使用指南
- 最佳实践建议

**命名规范：** `feature-name.md` 或 `how-to-xxx.md`

### 2. troubleshooting/ - 故障排除
问题诊断、错误修复和常见问题解答。

**适合放在这里的内容：**
- 错误诊断步骤
- 问题修复方案
- 常见问题 FAQ
- 调试技巧

**命名规范：** `issue-description.md` 或 `fix-xxx.md`

### 3. implementation/ - 实现总结
功能实现细节、技术决策和开发总结。

**适合放在这里的内容：**
- 功能实现说明
- 技术方案总结
- 配置变更记录
- 开发里程碑

**命名规范：** `feature-name.md` 或 `summary-xxx.md`

### 4. analysis/ - 技术分析
深度技术研究、性能分析和专家报告。

**适合放在这里的内容：**
- 技术深度分析
- 性能测试报告
- 安全研究文档
- 架构设计分析

**命名规范：** `topic-analysis.md` 或 `xxx-report.md`

## 添加新文档

1. **确定分类** - 根据文档内容选择合适的子目录
2. **创建文件** - 使用清晰的文件名（小写，用连字符分隔）
3. **更新索引** - 在 `docs/README.md` 中添加链接
4. **提交变更** - 使用描述性的 commit message

## 文档编写规范

### 标题层级
- 使用 `#` 作为文档标题（一级标题）
- 使用 `##` 作为主要章节（二级标题）
- 使用 `###` 作为子章节（三级标题）
- 避免使用四级及以下标题

### 代码块
使用语言标识的代码块：

\`\`\`typescript
// TypeScript 代码
\`\`\`

\`\`\`bash
# Shell 命令
\`\`\`

### 链接
- 使用相对路径链接其他文档
- 链接到代码文件时使用完整路径
- 外部链接使用完整 URL

### 语言
- 优先使用中文编写文档
- 技术术语可保留英文
- 重要文档可提供中英双语版本

## 文档维护

### 更新现有文档
- 保持文档与代码同步
- 及时更新过时信息
- 标注文档最后更新日期

### 删除文档
- 确认文档已过时且无参考价值
- 检查是否有其他文档引用
- 从 `docs/README.md` 中移除链接

### 重命名/移动文档
- 使用 `git mv` 保留文件历史
- 更新所有引用该文档的链接
- 在 `CHANGELOG.md` 中记录变更

## 示例

### 好的文档结构
```markdown
# 功能名称

简短的功能描述（1-2 句话）

## 使用场景

说明什么时候需要使用这个功能

## 配置步骤

1. 第一步
2. 第二步
3. 第三步

## 示例

提供实际的使用示例

## 常见问题

Q: 问题描述？
A: 解决方案

## 相关文档

- [相关文档1](link1.md)
- [相关文档2](link2.md)
```

### 提交信息示例
```
docs: 添加临时邮箱配置指南

在 docs/guides/ 中新增 temp-mail-config.md，
详细说明如何配置和使用临时邮箱功能。
```

## 注意事项

- **不要**在项目根目录创建新的 markdown 文件
- **不要**创建重复内容的文档
- **务必**在 `docs/README.md` 中更新索引
- **建议**在文档开头添加简短的摘要
- **推荐**使用清晰的标题和合理的章节划分
