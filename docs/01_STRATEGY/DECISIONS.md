# Business Decisions — ai-register
> 仅记录经用户明确确认的业务、范围、流程和体验决策。

## DEC-001 · 前端升级采用“产品化后台重构”路线
- **Date**: 2026-04-05
- **Decision**: 前端升级不采用渐进式工具页美化路线，改为基于 `React 18 + TypeScript + Vite` 的产品化后台重构。
- **Why**:
  - 用户明确要求使用现代前端栈重建界面。
  - 目标不只是替换技术栈，而是让当前配置、运行、日志、报告、临时邮箱相关能力进入统一控制台体验。
  - 参考图体现的是后台产品语言，不适合只做局部样式翻新。
- **Implication**:
  - 需要重新定义页面树、模块关系、组件系统和设计令牌。
  - 现有 Node/Playwright API 先作为后端能力层复用，前端优先重构为单独应用。
  - 后续 PRD、设计稿和实现默认以“内部测试控制台产品”视角展开。

## DEC-002 · 信息架构采用 Resource-Centric Operations Hub
- **Date**: 2026-04-05
- **Decision**: 控制台信息架构采用资源中心型后台，而不是围绕单次运行组织的 Run-Centric 工作台。
- **Why**:
  - 产品后续目标是自动化测试平台，而不只是本地测试控制面板。
  - 平台需要支持根据不同站点制定不同测试方案，这要求“站点”和“测试方案”成为长期核心资源。
  - 未来扩展重点会从“盯一次运行”转向“管理多站点、多方案、多产物、多次执行”。
- **Implication**:
  - 一级导航优先体现 `Overview / Config Center / Runs / Artifacts / System` 或其等价结构。
  - 需要补出“站点”和“测试方案”两类核心实体，而不只定义运行记录。
  - `Artifacts` 应按平台资产视角设计，支持跨运行查看报告、日志、截图、trace、token 等结果产物。

## DEC-003 · 平台持久化采用本地 SQLite 统一归档仓
- **Date**: 2026-04-05
- **Decision**: 配置、任务记录和测试产物不再以分散文件目录作为长期真源，改为由本地 SQLite 统一承接；同时补充按任务下载归档包的能力。
- **Why**:
  - 用户明确提出“配置切换到 SQLite”“token 保存到数据库”“按任务下载产物”。
  - 平台已经朝多站点、多方案、多任务演进，继续依赖 `config/`、`output_tokens/` 和散落目录会让前端与 API 长期绑定文件结构。
  - 本地 SQLite 能在不引入远程后端的前提下提供稳定的资源模型、查询能力和归档边界。
- **Implication**:
  - `Site / Plan / Target Profile / Mail Config / Task / Case / Run / Artifact` 将围绕 SQLite 建模。
  - token、日志、summary、报告、trace、截图、视频等输出将进入统一归档层，前端默认通过数据库元数据访问。
  - `Artifacts` 除按运行查看外，还要支持按任务聚合下载。
  - 旧文件路径转为迁移导入源或兼容回退边界，而不是前端和控制台的长期主数据源。

## DEC-004 · 浏览器环境配置作为授权测试验收能力保留
- **Date**: 2026-04-06
- **Decision**: 项目保留“浏览器环境配置”能力，但该能力仅用于已授权站点测试中的环境复现与审计，不得再以“设备指纹伪装”“反检测”或“绕过保护”为目标来定义。
- **Why**:
  - 用户明确说明这项能力是项目验收所必需，不能下线。
  - 项目需要验证的是：在组织批准的真实浏览器环境或模板下，授权站点的保护流程是否触发、是否拦截，以及人工处理后能否继续。
  - 只有将能力边界收敛到“可审计、可解释、可复现的测试环境配置”，才能同时满足业务验收与项目安全边界。
- **Implication**:
  - 允许项必须保留并进入验收清单：基于本地真实浏览器或组织批准模板导出并应用浏览器环境配置；支持 `UA / UA-CH / locale / timezone / viewport / screen / geolocation / 浏览器版本`；支持方案级绑定、来源追踪、审批标记、审计留痕和运行报告回显。
  - 禁止项必须保留并进入验收清单：`stealthMode`、Canvas/WebGL/Audio 噪声注入、插件伪造、`navigator` 深度篡改、随机轮换，以及“降低检测率”“规避机器人检测”“绕过保护”等目标表述。
  - `bot.sannysoft.com` 等检测站不得作为“有效性评分”主链路；验收必须以授权站点测试结果、配置回显和审计记录为主。
  - 后续所有涉及“指纹配置”的历史文档、字段、界面和 API，都需要逐步改写为“浏览器环境配置”或等价合规命名，并删除违规语义。
