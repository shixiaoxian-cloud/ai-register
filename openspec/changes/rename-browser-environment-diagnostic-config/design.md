## Context

当前仓库已经通过 `docs/00_MEMORY/CONTEXT_SNAPSHOT.md`、`docs/01_STRATEGY/DECISIONS.md` 和 `docs/02_PRD/framework-prd.md` 明确两件事：第一，浏览器环境配置是授权站点测试不可下线的正式验收能力；第二，这项能力只能服务于已授权测试中的环境复现与审计，不能再被实现为设备指纹伪装、反检测或绕过导向能力。与此同时，仓库内仍存在大量历史“指纹配置”资产，包括 `FINGERPRINT_README.md`、`docs/implementation/fingerprint-config-*.md`、`docs/guides/fingerprint-config-*.md`、`tools/fingerprint-exporter.html`、`tools/fingerprint-collector.js` 和 `config/fingerprint-presets/*.json`。这些资产中混杂了 `stealthMode`、Canvas/WebGL/Audio 噪声、检测站导入和“降低检测率”等旧语义，与当前项目安全边界冲突。

现有平台控制台已经具备 `Site / Plan / Profile / Mail Config` 的 SQLite 资源管理基础，但尚未把浏览器环境配置纳入平台资源，也没有把这项能力接入 `Plan` 绑定、运行前校验、运行详情回显和报告验收链路。运行时代码里虽然存在零散的“真实指纹”或“高级 stealth”逻辑，但它们不是基于可审计资源驱动，也没有和当前项目规则形成单一真值。

本次 change 必须解释清楚分层边界：
- 共享 baseline 继续负责平台级硬约束，例如显式配置、Fail Fast、统一日志、安全合规和测试回执。
- 当前仓库 overlay 继续负责项目目标与安全边界：只验证保护流程，不提供 CAPTCHA 绕过、短信验证绕过、设备指纹伪装或机器人规避实现。
- 本次新增的 project-owned capability 只定义“浏览器环境配置”在该仓库中的业务语义、运行边界、迁移治理和验收口径，不修改 baseline 的通用规则。

## Goals / Non-Goals

**Goals:**
- 将历史“指纹配置”统一收敛为“浏览器环境配置”，并形成一条受 OpenSpec 管控的项目私有 capability。
- 在平台 SQLite 模型、`Plan` 资源、前端配置中心和运行时链路中补齐浏览器环境配置的正式资源模型。
- 仅支持白名单字段：`UA`、`UA-CH`、`locale`、`timezone`、`viewport`、`screen`、`geolocation`、浏览器版本及其必要元数据。
- 为浏览器环境配置补齐来源追踪、审批标记、审计留痕和运行报告回显。
- 对历史模板、导出 JSON、文档、UI 文案和 API 命名执行完整治理，并提供迁移或受控淘汰路径。
- 让验收链路以授权站点测试结果、人工接续状态、配置回显和审计记录为主，而不是依赖检测站评分。

**Non-Goals:**
- 不实现 `stealthMode`、Canvas/WebGL/Audio 噪声注入、插件伪造、`navigator` 深度篡改、随机轮换等能力。
- 不为未授权站点访问、绕过保护、规避机器人检测或降低检测率提供任何实现。
- 不把 `bot.sannysoft.com` 等检测站重新引入为主流程 API、评分链路或必须依赖。
- 不在本次变更中扩展远程配置分发、多租户审批系统或复杂权限模型。

## Decisions

### 1. 以新的项目私有 capability 承载完整治理，而不是修改 baseline 规则

- Decision:
  - 本次变更新增三个项目私有 capability：`browser-environment-management`、`browser-environment-runtime-audit`、`browser-environment-governance`。
  - 不修改 `platform-core` 或 `spec-authoring` 的 baseline requirement，而是在项目能力层消费这些 baseline 约束。
- Why:
  - baseline 负责通用硬约束，当前问题是项目内一条具体能力的业务边界和历史债务收口，属于项目私有真值。
  - 这样既能保留共享规则的稳定性，也能把“浏览器环境配置”的细节写成可归档、可演进的能力规范。
- Alternatives considered:
  - 直接修改 `platform-core`：范围过大，会把仓库私有语义误升级为共享平台规则。
  - 只在产品文档中口头修正：缺少工程可执行的能力契约，无法支撑 proposal 之后的实现和归档。

### 2. 数据模型采用“白名单字段 + 审批元数据 + 审计快照”的资源结构

- Decision:
  - 新增 `BrowserEnvironmentConfig` 资源，持久化白名单字段、来源类型、来源摘要、审批状态、批准时间、版本信息和审计元数据。
  - `Plan` 显式绑定一个浏览器环境配置；配置被引用时禁止删除。
  - 历史 `fingerprint` 资产不再作为长期真值，只作为迁移输入。
- Why:
  - 只有把字段边界和审批/来源信息放进同一个资源模型，才能满足“可审计、可解释、可复现”的验收要求。
  - 运行时需要一份稳定的配置快照，而不是散落在文档、工具 JSON 和代码常量中的多套真值。
- Alternatives considered:
  - 继续沿用自由结构的 `config_json`：实现快，但无法确保白名单边界和审批信息被强约束。
  - 不在 `Plan` 上绑定，只在运行时临时选择：会破坏可复现性和报告回显的一致性。

### 3. 运行前使用受控适配层应用配置，而不是复用旧 stealth/fingerprint 逻辑

- Decision:
  - 在运行前新增一条浏览器环境配置适配层，只把白名单字段映射到 Playwright 允许的上下文设置、请求头或等价受控入口。
  - 旧 `stealth` / `advanced fingerprint` 逻辑不得作为浏览器环境配置的实现路径；相关代码要么被清理，要么被隔离出主链路。
  - 配置缺失、审批状态不满足、字段不一致或命中禁止项时，直接 Fail Fast，进入“未启动 / 启动前失败”状态，而不是半成功执行。
- Why:
  - 旧逻辑的设计目标偏向反检测和伪装，和当前能力定义冲突。
  - 只有通过受控适配层，才能明确“哪些字段能应用、哪些字段禁止应用”，并把失败点暴露为可审计错误。
- Alternatives considered:
  - 继续在 `src/stealth/*.ts` 上打补丁：最快，但会继续混淆“授权环境配置”和“规避型 stealth”两条语义。
  - 完全不复用现有运行时代码：风险过大，范围不必要扩大。

### 4. 报告与运行详情都必须回显“请求配置快照 + 实际生效摘要 + 审批信息”

- Decision:
  - 每次运行都记录三类信息：方案绑定的浏览器环境配置标识、执行前通过校验的配置快照、实际生效的白名单字段摘要。
  - 运行详情页、报告摘要和归档元数据必须显示配置来源、审批状态、版本、方案绑定关系和人工接续状态。
  - 验收摘要以“授权站点结果 + 人工接续结果 + 配置回显 + 审计记录”为主。
- Why:
  - 这项能力的核心不是“配置能保存”，而是“执行时到底用了什么环境，以及这是否满足授权测试验收”。
  - 仅记录原始日志不足以支撑后续复盘与审计，也难以回答客户验收问题。
- Alternatives considered:
  - 只在数据库里存配置，不在报告里回显：审计链路不完整。
  - 只展示请求配置，不展示实际生效摘要：用户无法确认运行时适配是否偏离了预期。

### 5. 历史资产采用“可映射即迁移，不可映射即阻断”的治理策略

- Decision:
  - 对历史模板、导出 JSON 和相关配置，如果只包含白名单字段，则迁移到 `BrowserEnvironmentConfig` 并保留迁移来源。
  - 对包含禁止字段、禁止语义或不可映射内容的历史资产，阻断迁移并给出明确原因，不做静默裁剪。
  - 对 README、guide、implementation docs、OpenSpec 产物和控制台文案执行统一命名治理，保留必要的历史说明但必须显式标为迁移背景。
- Why:
  - 静默裁剪会制造“表面成功、实际失真”的迁移结果，违背平台核心规范中的 Fail Fast 要求。
  - 完整治理方案的价值就在于一次性清理历史语义债，避免以后新旧表述长期并存。
- Alternatives considered:
  - 保留旧名称，仅在实现上限制字段：短期看省事，但文档、API 和用户心智会持续冲突。
  - 直接删除所有历史资产：会丢失已有模板和用户上下文，不利于平滑迁移。

### 6. 检测站从“评分主链路”降级为“可选补充诊断背景”

- Decision:
  - 正式验收、运行结果和报告评分不再以检测站结果为依据。
  - 若仓库需要保留检测站相关历史文档或人工排障说明，只能以补充诊断材料存在，不得以“评分 / 通过标准 / 推荐主流程”呈现。
- Why:
  - 当前能力已经明确服务于授权站点测试与环境复现；检测站评分既不稳定，也和当前安全边界不对齐。
  - 这条决策能避免后续实现误把“过检测站”当成目标。
- Alternatives considered:
  - 保留检测站评分作为辅助分数：容易再次滑回“反检测”目标语义。
  - 完全删除所有检测站历史说明：会丢失部分排障背景，可保留但必须降级。

## Risks / Trade-offs

- [Risk] 旧“指纹配置”文档和工具数量较多，仓库内可能长期残留旧语义。 -> Mitigation: 用统一检索规则做文档与代码审计，把允许保留的历史说明限制在迁移上下文里。
- [Risk] 现有运行时代码中已经混入 `stealth` / “真实指纹”实现，清理时容易误伤仍然需要的浏览器环境字段应用。 -> Mitigation: 先建立白名单适配层，再逐步把旧逻辑从主链路剥离。
- [Risk] 历史模板和导出 JSON 可能包含大量白名单外字段，导致迁移阻断比预期更多。 -> Mitigation: 提供迁移报告和阻断原因，让操作者知道哪些字段需要人工重建或放弃。
- [Risk] 地理位置、时区、UA-CH 等字段在不同浏览器版本上的实际可应用边界不完全一致。 -> Mitigation: 在运行前校验中显式记录哪些字段成功应用、哪些字段被拒绝，并在报告中回显。
- [Risk] 方案 B 涉及代码、文档、命名和迁移的全链路治理，写集较大。 -> Mitigation: 按“文档真值 -> 数据模型/API -> 前端 -> 运行时 -> 报告/迁移 -> 审计”波次推进，并保持统一验证门禁。

## Migration Plan

1. 先建立 OpenSpec 能力真值和统一命名，锁定允许项、禁止项、迁移边界和验收口径。
2. 在 SQLite 层新增浏览器环境配置模型，并为历史 `fingerprint` JSON / preset 定义迁移输入适配器。
3. 扩展 `Plan` 绑定和后端 API，先确保配置能被安全地保存、读取、校验和阻断。
4. 在前端配置中心增加浏览器环境配置资源页和方案绑定 UI，并隐藏或改写旧“指纹”入口。
5. 在运行前链路引入白名单适配层和 Fail Fast 校验，再把结果回显到运行详情、报告和产物元数据。
6. 最后做仓库级文档、文案、模板和历史资产治理，保留必要迁移说明，移除违规语义。

Rollback strategy:
- 迁移阶段不立即删除历史 JSON 和文档入口，而是先把它们标记为迁移来源或弃用入口。
- 若新资源模型或运行前校验导致回归，可暂时回退到“仅保存但不绑定执行”的中间状态，但不得重新开放禁止字段或违规语义。

## Open Questions

- 浏览器环境配置的审批元数据最小集合是否只需要 `approved`, `approvedBy`, `approvedAt`，还是还要补充模板责任人/审批单号。
- `geolocation` 在报告中应该回显到什么粒度，既满足审计，又不引入不必要的敏感展示风险。
- 对旧 `tools/fingerprint-exporter.html` 和 `tools/fingerprint-collector.js` 是直接重写为浏览器环境导出工具，还是保留文件名但彻底改内容；当前建议优先改语义和输出结构，再决定是否改文件名。
