## Context

当前控制台已经具备资源型后台的页面树，但共享壳层与页面骨架仍偏“工具页堆叠”：侧栏、顶栏和页面头部存在信息重复，`detail-layout` 在桌面端仍是单列，许多页面虽在文案上宣称“左侧台账、右侧详情”，实际却需要用户沿纵向长页面来回滚动查找。与此同时，危险操作仍通过原生 `window.confirm` 完成，导致样式、交互、可访问性和操作节奏完全脱离项目内设计体系。

这次变更沿用共享 baseline 和仓库 overlay 的分层边界：baseline 继续约束显式配置、安全边界、中文说明、测试留痕和工业级设计；仓库 overlay 则约束平台控制台应服务于授权测试、人工接续观察和安全合规，不得把 UI 重构扩展成任何绕过保护的行为设计。实现范围集中在前端壳层、样式系统、共享弹层和关键工作台页面，不修改保护验证链路、不改变日志语义和 API 真值来源。

## Goals / Non-Goals

**Goals:**
- 把平台控制台从“统一表格后台”提升为“资源中心型工作台”，强化首屏重点、主操作优先级和桌面端主次分区。
- 建立统一的 `Dialog / Drawer / Confirm` 组件与样式语义，替换原生确认框并收敛危险操作体验。
- 让 `Overview`、`Config Center`、`Task Center`、`Runs`、`System` 在共享布局规则下呈现一致的标题区、操作区、主列表区和详情/编辑区。
- 保留运行状态、日志、人工介入提示与产物跳转链路的可见性，使 UI 升级不削弱授权测试观察能力。
- 保证移动端与窄屏下仍可用，桌面端优先展现工作台布局。

**Non-Goals:**
- 不改动资源 API 语义、数据库模型或运行引擎。
- 不引入新的外部组件库。
- 不把详情编辑全部迁成弹层；本轮只优先覆盖高频新建/删除与更适合抽屉的入口。

## Decisions

### 1. 用“工作台壳层 + 页面编排”替代当前的单一长页堆叠

- Decision:
  - 保留现有路由和业务逻辑，但重做 `AppShell`、`WorkspaceHeader`、`SectionCard` 和全局容器样式。
  - 桌面端优先采用“主内容区 + 辅助区”或“列表区 + 详情区”的双栏结构；移动端降级为单列。
- Why:
  - 当前视觉问题主要不是控件单点不好看，而是布局主次没有真正兑现。
  - 在不改业务模型的前提下，先把页面编排收敛，能最大化改善整体产品感。
- Alternatives considered:
  - 只细调颜色和间距：不够，无法解决主次结构问题。
  - 全量重写每个页面：成本高，而且会放大与现有未提交改动的冲突面。

### 2. 弹层体系采用“确认弹窗 + 轻量模态框 + 侧滑抽屉”三件套

- Decision:
  - 新建共享 `DialogLayer`、`ConfirmDialog`、`DrawerPanel` 组件。
  - 删除等危险操作统一用确认弹窗；高频编辑入口优先用抽屉；较小的说明性或快捷操作使用模态框。
  - 所有弹层统一支持遮罩点击关闭、ESC 关闭、右上角关闭按钮、焦点样式与动画节奏。
- Why:
  - 当前最大体验断点是原生确认框与项目样式完全割裂。
  - 抽屉适合承接不离开上下文的编辑动作，比继续把整页拉长更符合工作台语义。
- Alternatives considered:
  - 只做确认弹窗：解决不了“便捷使用弹窗样式”的诉求。
  - 全部都做成模态框：复杂编辑会压缩内容空间，不利于配置页和任务页。

### 3. 样式收敛优先走全局 token 和共享 class，不在页面里继续分散硬编码

- Decision:
  - 继续以 `frontend/src/styles.css` 作为统一样式入口，升级颜色、层级、阴影、圆角、操作栏、表格、空状态和弹层 token。
  - 页面通过更稳定的共享 class 组合来表达布局，不继续在各页面散落新增独立视觉语义。
- Why:
  - 当前工程已经是集中式 CSS；在本轮不引入新组件库的前提下，继续集中收敛成本最低。
  - 先收敛 token 和共享块，能让后续页面继续升级时保持统一。
- Alternatives considered:
  - 每页局部补样式：会继续放大漂移。
  - 现在切到 CSS-in-JS 或 Tailwind：超出本轮范围。

### 4. 保持运行观察链路和人工接续口径不变，只升级呈现方式

- Decision:
  - 运行状态、阶段日志、人工介入提示、任务产物跳转仍沿用现有数据。
  - UI 只重构它们的布局层级、状态表达和交互承接方式，不改变真实业务判断。
- Why:
  - 项目价值在于观察保护是否触发、是否阻断以及人工完成后能否继续，不能因为重做界面而丢失链路。
  - 这也保证本轮主要风险集中在前端展示，而不是业务语义。

### 5. 写集边界以共享壳层和关键页面为主，避免碰后端和其他 change 的真值文件

- Decision:
  - 本次只在 `frontend/src/components/`、`frontend/src/pages/`、`frontend/src/styles.css` 做主要修改。
  - `frontend/src/lib/api.ts`、`frontend/src/lib/types.ts` 仅在实现弹层或页面布局确有需要时最小调整，否则保持只读。
  - 不修改现有已完成 change 的 OpenSpec 文档和后端平台存储逻辑。
- Why:
  - 当前工作树本来就有未提交变更，控制写集有助于减少误伤。
  - 这次目标明确是 UI 与交互跃迁，不应顺手扩大到资源模型层。

## Risks / Trade-offs

- [Risk] `Config Center` 和 `Task Center` 文件体量已经较大，继续在原文件里调整布局会增加维护压力。 -> Mitigation: 本轮优先抽共享弹层和局部布局块，避免无边界追加逻辑；如发现继续膨胀，再单独拆分。
- [Risk] 抽屉与模态框切入现有页面后，局部状态管理可能变复杂。 -> Mitigation: 仅把高收益入口迁入弹层，并保持表单状态仍由原页面控制。
- [Risk] 全局样式重构容易影响所有页面。 -> Mitigation: 先保持 class 名兼容，在原语义上升级 token 和布局，而不是一次性推翻所有命名。
- [Risk] 工作区已有未提交样式与页面改动，可能与本轮变更重叠。 -> Mitigation: 实现前逐个读取受影响文件，以当前工作树为准合并，不回退他人改动。
- [Risk] 弹层动画和遮罩如果处理不好，会带来可访问性和滚动问题。 -> Mitigation: 统一在共享组件层处理 `role`、`aria-modal`、ESC、overlay click 和 body scroll 锁定。

## Migration Plan

1. 先建立共享工作台样式和弹层基础组件，保证所有页面后续都能复用。
2. 再升级 `AppShell` 和页面头部，统一导航、顶栏、操作区和内容容器。
3. 按优先级重构 `Overview`、`Config Center`、`Task Center`、`Runs`、`System` 的布局和主次关系。
4. 把 `Config Center` 与 `Task Center` 的删除确认替换成统一确认弹窗，并在高频入口上补充抽屉/模态承接。
5. 执行 `npm run console:typecheck` 与 `npm run console:build`，再记录优化日志。

Rollback strategy:
- 若某个页面布局改动引发明显回归，可以保留共享弹层和 token 收敛，只回退对应页面编排。
- 若弹层交互存在阻塞问题，可先保留确认弹窗替换，延后非关键抽屉入口。

## Open Questions

- `Config Center` 后续是否要继续拆文件，当前先按“局部共享块抽离 + 页面内整合”处理。
- `Artifacts` 本轮是否同步进入更强的工作台布局，当前优先级次于首页、配置、任务、运行和系统页；若共享样式带来的提升已足够，可只做轻量跟进。
