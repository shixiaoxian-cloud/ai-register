## ADDED Requirements

### Requirement: 平台控制台必须提供统一的模态框与抽屉组件
平台控制台 MUST 提供共享的 Modal 与 Drawer 交互原语，用于承接快捷操作、上下文编辑和补充信息展示。所有弹层 MUST 使用统一的遮罩、层级、圆角、间距、动画时长和关闭机制，不得继续依赖浏览器原生确认框或页面内随意拼接的临时遮罩实现。

#### Scenario: 页面触发需要保留上下文的编辑动作
- **WHEN** 用户在工作台页面中发起适合就地完成的编辑或查看动作
- **THEN** 系统应优先使用统一抽屉或模态框承接该动作
- **AND** 用户关闭弹层后应仍能保留原页面上下文

#### Scenario: 弹层打开与关闭
- **WHEN** 任一模态框或抽屉被打开或关闭
- **THEN** 系统必须使用统一的淡入、缩放或滑入动画节奏
- **AND** 系统必须支持遮罩点击关闭、关闭按钮关闭和 ESC 关闭，除非当前操作被明确设计为不可中断

### Requirement: 危险操作必须通过统一确认弹窗完成
平台控制台 MUST 对删除、移除或等价危险操作提供统一的确认弹窗，弹窗 MUST 明确当前操作对象、潜在影响、主次按钮和取消路径。系统 MUST 不得继续使用 `window.confirm` 或等价原生确认框来执行正式危险操作。

#### Scenario: 用户删除任务或配置资源
- **WHEN** 用户点击删除任务、站点、方案、画像、邮箱配置或等价危险操作
- **THEN** 系统必须打开统一确认弹窗并明确显示被删除对象名称
- **AND** 用户必须能够主动取消而不发生副作用

#### Scenario: 危险操作确认后执行
- **WHEN** 用户在确认弹窗中明确确认危险操作
- **THEN** 系统才可以继续执行对应删除或移除逻辑
- **AND** 若用户关闭或取消弹窗，系统不得执行该操作

### Requirement: 弹层必须满足可访问性和状态完整性要求
平台控制台的弹层 MUST 具备 `role`、`aria-modal` 或等价可访问性标识，并覆盖 loading、success、error、disabled 与 focus 状态。系统 MUST 不得让弹层只通过颜色表达状态，也不得在异步执行中缺少明确的等待或失败反馈。

#### Scenario: 弹层内存在异步确认动作
- **WHEN** 用户在确认弹窗或模态框中触发异步提交
- **THEN** 系统必须显示可感知的 loading 或 disabled 状态
- **AND** 成功或失败结果必须有明确反馈，而不是无提示关闭或停滞

#### Scenario: 键盘用户操作弹层
- **WHEN** 用户通过键盘与弹层交互
- **THEN** 关闭按钮、主操作按钮和关键输入项必须具备可见的 focus 状态
- **AND** 弹层本身必须具有可访问的语义标识
