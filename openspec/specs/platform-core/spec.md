# Platform Core Specification

## Purpose

本规范定义仓库级平台开发基线，约束所有实现必须遵守的配置管理、日志安全、工业级设计、中文编码与交付验证要求。任何 capability、设计或任务拆解都必须先满足这些基线，再讨论具体业务实现。

## Requirements

### Requirement: 显式配置与确定性约束

所有实现 MUST 以显式配置文件作为配置真值，并在变更前后检查垃圾配置、无效配置、重复配置、失效配置、冲突配置、未引用配置与无效默认值。配置校验失败时 MUST 立即失败，不得静默回退到隐式默认值。

#### Scenario: 新增或修改配置项

- **WHEN** 代理新增、删除或修改任意配置项
- **THEN** 代理必须同时说明配置来源、消费者、默认值、作用域、生效顺序与清理策略
- **AND** 代理不得引入 `process.env`、`os.environ`、`.env` 或 `dotenv` 作为绕过显式配置文件的第二真值
- **AND** 配置错误必须以 Fail Fast 方式暴露，而不是在运行时静默替换为不透明默认值

### Requirement: 统一日志与安全合规

所有实现 MUST 使用统一日志模块输出可观测信息，不得使用临时打印语句替代正式日志能力，也不得在代码或日志中泄漏敏感信息。

#### Scenario: 为关键路径补充诊断信息

- **WHEN** 代理需要为关键业务路径、失败链路或排障入口补充日志
- **THEN** 代理必须使用仓库统一日志模块，而不是 `print()`、`console.log()` 或 `fmt.Println()`
- **AND** 日志内容不得包含 `Secret`、`Token`、`Password`、`API Key` 或其他敏感信息明文
- **AND** 如需补充审计或调试信息，必须保证数据口径稳定且可追溯

### Requirement: 工业级设计覆盖

所有产品方案、编码方案和实现结果 MUST 达到工业级设计要求，覆盖主流程、异常流程、空状态、边界条件、权限口径、可观测性、性能、兼容性、回滚与审计要求，不得只描述理想路径。

#### Scenario: 设计或修改核心功能

- **WHEN** 代理分析、设计或实现核心功能、关键链路或高风险改动
- **THEN** 输出必须明确目标用户、输入输出、依赖关系、成功标准与非目标范围
- **AND** 必须覆盖失败路径、权限不足、数据缺失、配置异常、超时失败、重试恢复与兼容策略
- **AND** 必须优先修复根因，避免只做表面补丁或顺手改动无关逻辑

### Requirement: 后端用户输入入口必须执行格式、内容与合理范围校验

所有面向用户输入的后端创建、更新、执行、测试、导入或等价入口 MUST 在进入持久化、任务执行或下游副作用前执行格式校验、内容校验和合理范围校验。后端 MUST 不得因为前端已经做过本地校验就跳过最终输入校验，也不得把无效输入静默裁剪为看似成功的结果。

#### Scenario: 后端收到结构或格式非法的输入

- **WHEN** 任意后端入口收到字段缺失、类型不符、格式不合法或结构不完整的用户输入
- **THEN** 后端必须在执行业务副作用前拒绝该请求
- **AND** 后端不得以静默修正或隐式默认值掩盖该输入问题

#### Scenario: 后端收到内容或范围不合理的输入

- **WHEN** 任意后端入口收到语义上不允许的内容，或长度、数值、数量、时间区间、阈值等超出允许范围的用户输入
- **THEN** 后端必须在持久化或执行前阻断该请求
- **AND** 后端不得因为前端本地已经通过校验而跳过这类最终裁决

### Requirement: 后端校验失败必须返回正确状态码、明确 message 与可映射错误结构

后端在输入校验失败时 MUST 返回与失败语义匹配的正确状态码、稳定且可展示的 `message`，以及字段级错误、表单级错误或等价的可映射错误结构，使前端能够定位问题并展示可理解提示。对于已知的输入校验、唯一性冲突、资源状态冲突或等价可预期失败，后端 MUST 不得返回 `200`、`500` 或其他与语义不符的状态码，也不得把原始异常堆栈、底层库错误或不透明字符串直接暴露为面向最终用户的返回内容。

#### Scenario: 提交阶段触发格式、内容或范围校验失败

- **WHEN** 用户提交请求后，后端在格式、内容或合理范围校验阶段发现输入不合法
- **THEN** 后端必须返回与该失败语义匹配的 `4xx` 状态码和可直接展示的 `message`
- **AND** 若错误可定位到具体输入项，后端还必须返回字段级错误结构或等价信息

#### Scenario: 提交阶段触发唯一性或状态冲突

- **WHEN** 用户提交请求后，因为名称重复、并发写入、资源状态不允许或等价冲突而被后端拒绝
- **THEN** 后端必须返回与冲突语义匹配的状态码，而不是通用成功码或服务端异常码
- **AND** 返回的 `message` 必须明确说明当前为什么不能继续提交

#### Scenario: 底层异常参与输入校验失败链路

- **WHEN** 底层 schema、解析器或业务校验模块产生原始异常信息
- **THEN** 后端必须将其收敛为稳定的对外状态码、`message` 与错误结构后再返回
- **AND** 原始异常细节仅可进入日志、审计或调试链路，不得直接暴露给最终用户

### Requirement: 中文编码与关键注释一致性

所有新增或修改的中文文本文件 MUST 保持 UTF-8 编码；核心业务规则、复杂状态流转、关键算法、配置解析链路、迁移回滚逻辑以及性能或安全关键路径 MUST 维护高价值中文注释，并在修改实现时同步更新相关注释。

#### Scenario: 修改关键代码或中文文档

- **WHEN** 代理修改关键代码、复杂流程或任何中文文档内容
- **THEN** 代理必须检查文件、终端或页面显示是否出现乱码
- **AND** 对关键逻辑补充或更新说明“为什么这样做、边界在哪里、哪些点不能随便改”的中文注释
- **AND** 代理不得保留过期、失真或与实现不一致的注释

### Requirement: 交付验证与审计留痕

所有交付在完成实现后 MUST 优先执行可自动化的测试、构建或验证命令，并保留测试回执；大规模修改还 MUST 执行一次审计或一致性检查，以保证变更可回归、可审计、可追溯。

#### Scenario: 完成一次实现交付

- **WHEN** 代理完成编码并准备交付结果
- **THEN** 代理必须先执行本次改动可运行的测试、构建或验证命令
- **AND** 若存在无法自动执行的项目，必须明确记录未执行项、阻塞原因、替代验证方式与后续动作
- **AND** 若本次改动属于 3+ 文件或 100+ 行的大规模修改，代理必须补充一次审计或一致性检查结论

### Requirement: 一次性运维脚本与自动化测试必须按目录职责落位

仓库内新增或修改的一次性迁移、回填、巡检、重建、基准与人工运维脚本 MUST 落在 `scripts/`；由测试框架执行、以断言给出通过/失败结论的单元测试、集成测试、回归测试与专项自动化验证 MUST 落在 `tests/`。`app/` 不得新增以一次性迁移或自动化测试为主职责的入口文件。

#### Scenario: 新增一次性迁移或回填入口

- **WHEN** 代理新增数据库迁移、历史数据回填、巡检修复、重建或其他一次性运维入口
- **THEN** 对应入口文件必须放在 `scripts/`
- **AND** 不得把这类入口直接新增到 `app/`、`frontend/` 或其他运行时代码目录

#### Scenario: 新增自动化断言型验证

- **WHEN** 代理新增由 `pytest` 或等价测试框架执行的单元测试、集成测试、回归测试或专项自动化验证
- **THEN** 对应测试文件必须放在 `tests/`
- **AND** 不得把这类自动化测试入口放在 `scripts/` 并让其承担测试目录职责

#### Scenario: 新增人工运维校验脚本

- **WHEN** 代理新增人工触发、环境相关、可能带副作用的运维校验工具
- **THEN** 该工具可以放在 `scripts/`
- **AND** 系统不得因为文件名包含 `verify` 就把它误判为必须进入 `tests/` 的自动化测试

### Requirement: 运行时代码与脚本/测试目录必须保持单向依赖

`scripts/` 与 `tests/` 可以依赖 `app/` 中的运行时或可复用业务模块；但 `app/` MUST 不得反向依赖 `scripts/` 或 `tests/`。若脚本或测试需要复用通用逻辑，只允许把真正稳定、与一次性迁移/测试语义无关的纯业务模块抽取到 `app/`，而不得把一次性主流程继续保留在 `app/` 中。

#### Scenario: 脚本需要复用业务逻辑

- **WHEN** 某个 `scripts/` 入口需要复用运行时共用的数据访问、校验或领域逻辑
- **THEN** 代理可以让 `scripts/` 依赖 `app/` 中稳定且通用的业务模块
- **AND** 不得因此把一次性迁移或运维主流程整体迁回 `app/`

#### Scenario: 运行时代码引用脚本或测试目录

- **WHEN** 代理修改 `app/` 下的运行时代码
- **THEN** 运行时代码不得 import `scripts/` 或 `tests/`
- **AND** 若出现这类依赖，代理必须在交付前清理或显式记录阻塞原因

### Requirement: 后端运行时代码单文件行数必须受控

`app/` 下由团队维护的后端运行时代码 MUST 遵守单文件物理行数上限：新增或重写后的单个源码文件必须不超过 800 行。对于已经超过 800 行的存量文件，后续改动 MUST 优先拆分职责并向阈值收敛，不得在无说明的情况下继续把更多功能堆入同一文件。

#### Scenario: 新增后端运行时代码文件

- **WHEN** 代理或开发者在 `app/` 下新增由团队维护的 Python 运行时代码文件
- **THEN** 该文件落盘后的物理行数必须不超过 800 行
- **AND** 文件职责必须保持单一，不得通过把无关分支继续堆叠在同一文件内绕过约束

#### Scenario: 修改已超限的后端存量文件

- **WHEN** 代理或开发者修改一个已经超过 800 行的 `app/` 运行时代码文件
- **THEN** 本轮改动必须优先收敛该文件职责，或至少不得在无说明的情况下继续增加文件总行数
- **AND** 若受回归风险、生成链路或交付窗口限制暂时无法降到阈值内，必须在对应 change 产物与交付日志中记录临时例外、拆分计划与后续目标

#### Scenario: 超限文件属于自动生成或等价镜像产物

- **WHEN** 某个超过 800 行的后端运行时代码文件来自稳定生成链路或等价镜像过程，拆分会破坏生成真值
- **THEN** 仅可在仓库已有可追溯生成来源、且 change 产物与交付日志显式说明原因时保留临时例外
- **AND** 不得把“生成文件”作为人类维护超大文件的通用豁免理由

### Requirement: 单文件行数统计口径必须唯一且可审计

仓库内单文件行数统计 MUST 以版本库中文件保存后的物理行数为唯一真值，包含空行、注释、import 区与普通代码行。人工 review、优化日志、脚本审计与方案说明 MUST 使用同一统计口径，不得在同一仓库内同时维护第二套“有效代码行”或 IDE 私有统计口径。

#### Scenario: 人工核对或脚本审计单文件行数

- **WHEN** 代理、开发者或审计脚本需要判断某个文件是否超过单文件行数阈值
- **THEN** 必须使用文件落盘后的物理行数作为唯一统计结果
- **AND** 不得在说明、审计或验证阶段切换为另一套不兼容的统计口径

### Requirement: 平台级公共包入口导出的运行时对象必须在同名子模块存在时保持稳定对象语义

当平台级公共包入口对外承诺导出某个运行时对象，且包内同时存在同名子模块时，系统 MUST 保证 `from <package> import <symbol>` 返回的是该运行时对象或其等价惰性代理，而不是误把同名子模块暴露给调用方。系统不得因为 import 阴影导致调用方在运行期失去既有方法访问能力。

#### Scenario: 调用方通过包入口导入 scheduler 运行时对象

- **WHEN** 调用方执行 `from app.platform.scheduler import scheduler` 并继续调用 `add_job()`、`get_jobs()` 或 `remove_job()`
- **THEN** 系统必须把调用转发到真实 scheduler 运行时对象
- **AND** 不得因为同名 `scheduler.py` 子模块存在而让调用方拿到模块对象

#### Scenario: 包入口仍需保持轻量导入

- **WHEN** 调用方只导入 `TaskRegistry`、`ExecutionContext` 或其他轻量符号
- **THEN** 系统仍可继续保持 scheduler runtime 的惰性加载
- **AND** 不得为了修复同名导出问题而把所有轻量导入路径强制升级为完整 scheduler runtime 导入
